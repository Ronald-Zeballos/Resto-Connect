Quiero que analices y refactorices/generes mi backend siguiendo una arquitectura por paquetes orientada a casos de uso, no por carpetas técnicas genéricas.

Contexto:
Tengo un sistema llamado RestoConnect Pro para restaurantes. Actualmente la estructura está muy parecida a un backend .NET organizado por módulos como:
- Almacen
  - GestionAlmacenes
  - Inventario
  - SalidasStock
- Compras
  - Catalogo
  - EspecificacionesCompra
  - Proveedores

  - FlujoCaja
  - PRC
  - Presupuesto
  - TIR
- Infraestructura
- Interfaces
- Migraciones
- Proyectos

Pero me indicaron que la arquitectura debe estar organizada “por paquetes tipo cada caso de uso debe tener su propio paquete para tener esa funcionalidad”. Es decir, no quiero una estructura donde todos los controllers estén juntos, todos los services juntos y todos los DTOs juntos. Quiero que cada funcionalidad tenga su propio paquete autocontenido.

Tecnología objetivo:
Usar Java 21 con Spring Boot 3.
Usar Maven.
Usar PostgreSQL.
Usar Spring Data JPA.
Usar Bean Validation.
Usar Lombok si ayuda a reducir código.
Usar MapStruct si es necesario para mapear entidades a DTOs.
Usar Spring Security con JWT básico para roles ADMIN, MESERO, COCINA y CLIENTE_QR.
Usar Flyway o Liquibase para migraciones.
Usar WebSocket o Server-Sent Events para notificaciones en tiempo real.
Preparar el código para poder correr con Docker Compose: backend + PostgreSQL.

Objetivo principal:
Reestructurar el proyecto para que cada caso de uso tenga su propio paquete con:
- Controller
- Service o UseCase
- DTOs
- Entidades relacionadas si aplica
- Repository
- Validaciones
- Eventos
- Tests básicos

No quiero una arquitectura superficial. Quiero que la lógica de negocio sea real y mejorada, especialmente en inventario, compras, predicción de reposición con IA/reglas inteligentes y notificaciones.

No quiero un inventario básico donde solo se suma y resta stock. Quiero un inventario más parametrizable e inteligente.

El módulo de inventario debe manejar:
- Ítems de inventario.

- Tiempo promedio de entrega del proveedor en días.

Ejemplo:

- 1 pan
- 1 carne

- 1 lámina de queso

Cuando se vende una hamburguesa, el sistema debe descontar automáticamente esos insumos del inventario.

Lógica de negocio requerida para inventario:
1. Al crear un pedido confirmado, descontar automáticamente los insumos relacionados a cada producto.
2. Si el stock llega a cero, marcar el producto del menú como no disponible.
3. Si el stock baja del mínimo, generar alerta.
4. Si el stock baja del punto de reorden, sugerir compra.
5. Si hay sobrestock, generar advertencia.
6. Si el consumo diario aumenta, ajustar la predicción.
7. Si un proveedor demora más días, aumentar cantidad sugerida.
8. Si hay historial insuficiente, usar reglas simples.
9. Si hay historial suficiente, usar predicción basada en promedio móvil.
10. Dejar preparado un adaptador para IA externa, pero implementar primero un motor local de reglas y promedio móvil.

Predicción de reposición:
Crear un paquete inventario.prediccion con un motor de predicción.

Debe calcular:
- consumoPromedioDiario
- diasHastaAgotamiento
- fechaEstimadaAgotamiento
- cantidadSugeridaCompra
- nivelRiesgo: BAJO, MEDIO, ALTO, CRITICO
- motivo de la recomendación
- confianza de la predicción en porcentaje

Fórmula sugerida:
consumoPromedioDiario = promedio de salidas de inventario de los últimos N días.
diasHastaAgotamiento = stockActual / consumoPromedioDiario.
cantidadSugeridaCompra = consumoPromedioDiario * (diasEntregaProveedor + diasCoberturaDeseada) - stockActual.
Si cantidadSugeridaCompra < 0, devolver 0.

Reglas importantes de negocio:
- No permitir crear pedido si una mesa está inactiva o bloqueada.
- No permitir confirmar pedido si no hay stock suficiente de los insumos.
- Al confirmar pedido, descontar inventario mediante movimientos de salida.
- Si falla el descuento de inventario, no se debe confirmar el pedido.
- Usar transacciones con @Transactional.
- No permitir stock negativo.
- No permitir eliminar categorías con productos activos.
- No permitir eliminar proveedor con órdenes activas.
- El pago electrónico puede estar simulado, pero debe tener estructura preparada para integrar una pasarela real.
- La factura se genera solo si el pedido está pagado.
- En efectivo, el pedido puede quedar pendiente de pago hasta que el mesero confirme.
- Las notificaciones deben generarse como eventos de dominio o llamadas desde los casos de uso.
- Mantener separación entre lógica de negocio y controlador.
